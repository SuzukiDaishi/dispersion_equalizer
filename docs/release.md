# Release Guide

This project publishes builds through GitHub Actions.

Normal branch pushes and pull requests run CI and upload short-lived build
artifacts. Public GitHub Releases are created only when a `v*` tag is pushed.

## Release Checklist

1. Make sure the working tree is clean.

   ```powershell
   git status
   ```

2. Run the local checks.

   ```powershell
   cargo check
   cargo test
   cargo xtask bundle dispersion_equalizer --release
   ```

3. Update `version` in `Cargo.toml` to the new version, then commit.
   This is the **only place** to change the version — `Plugin::VERSION` reads
   `CARGO_PKG_VERSION` at compile time, so everything else follows automatically.

   ```powershell
   git add Cargo.toml Cargo.lock
   git commit -m "Release v0.1.5"
   ```

4. Push the commit.

   ```powershell
   git push
   ```

5. Create and push a release tag. The tag version **must match** `Cargo.toml`
   exactly — the release workflow enforces this and will fail otherwise.

   ```powershell
   git tag v0.1.5
   git push origin v0.1.5
   ```

6. Open GitHub Actions and wait for the `Release` workflow to finish.

7. Confirm the GitHub Release has these assets:

   ```text
   Dispersion.Equalizer-v0.1.5-windows-x64.zip
   Dispersion.Equalizer-v0.1.5-linux-x64.zip
   Dispersion.Equalizer-v0.1.5-macos-universal.dmg
   ```

8. Download at least one archive and confirm it contains:

   ```text
   Dispersion Equalizer.vst3
   Dispersion Equalizer.clap
   ```

## Notes

- Tags must use the `v*` format, e.g. `v0.1.5`.
- The release workflow verifies that the git tag and `Cargo.toml` version match.
  Pushing a tag that differs from `Cargo.toml` causes the workflow to fail immediately.
- Release notes are generated by GitHub.
- Commit builds are stored as Actions artifacts for 7 days.
- Release builds target Windows x64, Linux x64, and macOS Universal Binary (arm64 + x86_64).
- macOS builds are Developer ID signed and notarized via Apple notarytool.
